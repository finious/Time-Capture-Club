# THE HAND WE ARE PLAYING

We are here to make the work undeniable.

Time-Capture-Club is not another AI website generator.

It is the operating practice that begins after the thrilling sentence:

> “The site works.”

That sentence is not the end of the job.

It is the moment the deeper questions begin:

- Where is the source of truth?
- Which version is live?
- What changed?
- Why did it change?
- What was tested?
- What remains unknown?
- Where is the rollback?
- Can the owner return tomorrow without reconstructing the project from memory?

AI can help a person make a website quickly.

Time-Capture-Club helps that person become able to understand, govern, test, release, recover, and return to the work.

That is the product.

# THE PROBLEM WE PAID TO DISCOVER

We did not invent this problem from a market-research worksheet.

We paid for it with our own mistakes.

Across three live Codex-built websites, we learned that a project can be online while its owner still lacks command of it.

We:

- celebrated successful ZIP deployments,
- moved faster than our documentation,
- blurred source, archive, export, and deploy cargo,
- treated host confirmation as proof,
- discovered broken or uncertain release paths,
- and had to reconstruct decisions that should have been preserved.

The governing realization was:

> **We created a monster and did not use the original lessons of structured files.**

The second realization was:

> **A visible result can hide an invisible operating failure.**

Those failures became tuition.

Time-Capture-Club gives the tuition away.

> **You should not have to repeat our mistakes to launch and govern a professional website.**

The product must not hide GitHub behind a false magic button.

The user should understand the release rail:

> **Local source → reviewed change → meaningful commit → GitHub push → host deployment → live smoke test → release receipt**

Each step has a different job.

A commit is not a deployment.

A push is not a verified release.

A successful host message is not proof that the experience works.

A live homepage is not proof that the form submits.

A ZIP is deployment cargo.

It is not the source, the archive, the operating system, or the home of the work.

Protected ruling:

> **We do not move into the shipping crate.**

# THE COMPLETE LOOP

The Build Week prototype must demonstrate one complete professional loop:

> **Vision → Structure → GitHub → Change Review → Smoke Test → Release Gate → Release Receipt → Return**

## 1. Vision

Capture:

- site name,
- purpose,
- audience,
- promise,
- primary action,
- live URL,
- and current condition.

## 2. Structure

Establish clear roles for:

- live source,
- content,
- assets,
- documentation,
- tests,
- archive,
- and deploy-ready cargo.

Exact folder names may vary.

Folder roles may not remain ambiguous.

## 3. GitHub

Establish:

- repository,
- production branch,
- commit practice,
- remote connection,
- host connection,
- and last known good commit.

## 4. Change Review

Show:

- files added,
- files changed,
- files removed,
- risky paths,
- routing changes,
- form changes,
- configuration changes,
- and human-readable intent.

## 5. Smoke Test

Use only:

- PASS
- FAIL
- NOT VERIFIED
- NOT APPLICABLE

Check the live experience, including:

- homepage,
- mobile layout,
- navigation,
- primary action,
- forms,
- thank-you route,
- internal links,
- external links,
- apex domain,
- `www` redirect,
- HTTPS,
- metadata,
- favicon,
- placeholders,
- deploy cargo,
- and correspondence between the intended commit and the live release.

No fake certainty.

## 6. Release Gate

The human chooses:

- RELEASE APPROVED
- RELEASE BLOCKED
- RELEASE APPROVED WITH KNOWN RISKS

Automation may provide evidence.

It may not erase judgment.

## 7. Release Receipt

Export a portable Markdown record containing:

- project,
- date,
- commit,
- change summary,
- reason,
- test results,
- failures,
- unknowns,
- known risks,
- live destination,
- rollback point,
- reviewer,
- and next action.

## 8. Return

Show:

- current status,
- last verified release,
- open risks,
- recent receipts,
- last known good commit,
- and next honest action.

Then release the user from dashboard gravity.

The goal is return to the real work.

# THE SITE OPERATING CONSTITUTION

Every site governed through Time-Capture-Club follows these laws.

## Law 1: The live site is not the source of truth

The verified source project and repository are the recognized source.

## Law 2: ZIP is cargo, not home

We do not move into the shipping crate.

## Law 3: Every project has an identity card

It records:

- purpose,
- audience,
- local source,
- repository,
- production branch,
- live URL,
- host,
- publish directory,
- last verified release,
- known risks,
- and next action.

## Law 4: Every project has one explainable release rail

The owner must be able to describe how local work becomes live work.

## Law 5: Every test state is honest

PASS, FAIL, NOT VERIFIED, or NOT APPLICABLE.

## Law 6: Every meaningful release leaves a receipt

No release becomes a disappearing event.

## Law 7: Every code session can leave a human review packet

The machine explains what it changed.

## Law 8: Production represents approved work

Experiments do not silently become production truth.

## Law 9: Deployment never replaces verification

The live experience must be checked.

## Law 10: Every session ends with the next honest action

No green badge may substitute for understanding.
